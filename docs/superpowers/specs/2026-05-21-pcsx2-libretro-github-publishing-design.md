# pcsx2-libretro GitHub publishing — design

**Status:** Spec for implementation
**Date:** 2026-05-21
**Repos touched:** `pcsx2-libretro` (this brainstorm); RetroNest-Project untouched
**Sub-project label:** SP-publish (one-off, not in the SP1–SP10 main arc)

## Problem

The pcsx2-libretro fork at `/Users/mark/Documents/Projects/pcsx2-libretro/` is local-only. Its sole git remote is `upstream` (PCSX2/pcsx2). The built `pcsx2_libretro.dylib` and its sibling `pcsx2_libretro_resources/` are deployed by hand into `~/Documents/RetroNest/emulators/libretro/cores/` after every rebuild.

For RetroNest to ship as a real application, the core needs a public download location. The existing `PatchesInstaller` pattern (fetches `patches.zip` from `github.com/PCSX2/pcsx2_patches/releases/latest`) establishes the precedent: GitHub Releases as the artifact source, staleness-gated.

This spec covers the publishing-side infrastructure only. The RetroNest-side installer that consumes the published artifacts is a separate future sub-project.

## Goals

1. Push the fork to a public GitHub repo under the user's account so the source is visible and reproducible.
2. Build the core in CI on tag push, package with resources + version metadata, and attach to a GitHub Release.
3. Document the upstream-rebase + release process in the repo so it survives loss of working memory.

## Non-goals

- RetroNest-side installer (separate sub-project).
- Windows or Linux builds (Windows is a separate future sub-project; Linux has no current consumer).
- Universal arm64+x86_64 macOS binary — x86_64-only is chosen below; rationale in Decisions.
- Code signing / notarization (libretro cores aren't gatekeeper-checked when loaded by a notarized host).
- Pushing pre-existing upstream PCSX2 release tags (`v2.7.95`–`v2.7.99`) — they'd visually tangle with the fork's `v2026.*` tags.

## Decisions

**D1. Repository scope: full fork including upstream PCSX2 history.**
Push the entire fork (~1 GB) as `main` to `github.com/markrpearce96/pcsx2-libretro`. Rejected alternatives: a slim libretro-only repo with PCSX2 as a submodule (every rebase becomes a submodule bump + manual conflict resolution); a release-only repo with source kept local (loses public reproducibility).

**D2. Build pipeline: GitHub Actions on tag push.**
Workflow runs only on `push: tags: ['v*']`. Zero CI cost between releases. The maintainer pushes a tag when they want a release; CI builds and publishes.

**D3. Architecture: macOS x86_64 only (not universal).**
Upstream PCSX2 has no AArch64 recompilers landed — Apple Silicon native builds fall back to interpreters at ~65–70% of vsync. The Rosetta x86_64 path (SP10) measured 3.1× faster (32.4% → 100% on R&C 2). The user's RetroNest already prompts a slow-mode toast for arm64 launches, signaling that x86_64-under-Rosetta is the intended runtime. Shipping x86_64-only halves CI complexity, eliminates the lipo step, and matches the actual recommended runtime path. A future native arm64 release becomes viable when (and if) upstream lands AArch64 recompilers.

**D4. Artifact: single zip `pcsx2_libretro-macos-x86_64.zip`.**
Contains `pcsx2_libretro.dylib`, `pcsx2_libretro_resources/`, and a `VERSION` file. Mirrors the patches.zip pattern that RetroNest's existing `PatchesInstaller` already handles. Platform suffix in the filename leaves clean room for future `-windows-x86_64` and `-macos-arm64` artifacts.

**D5. Tag scheme: date-based `vYYYY.MM.DD`.**
Releases cut whenever the maintainer wants. Multiple tags on one day get a numeric suffix (`v2026.05.21.1`). Date tags signal freshness and avoid inventing a semver policy for what is effectively continuous deployment of a fork.

**D6. Release notes: auto-generated by `softprops/action-gh-release@v2` with a small custom prelude.**
Prelude (platform + upstream SHA) lives inline in the workflow YAML next to where the SHAs are computed. No separate template file in the repo.

## Architecture

### Repo layout (delta from current state)

```
pcsx2-libretro/                         (existing fork root)
├── .github/workflows/
│   ├── libretro_release.yml           NEW — this spec
│   └── ... (existing upstream PCSX2 workflows, untouched)
├── README.md                          NEW — fork intro, install pointer, build quickstart, GPL-3.0 note
├── UPSTREAM-UPDATE.md                 NEW — rebase + release process
├── pcsx2-libretro/                    (existing — source dir, untouched)
└── ... (rest of upstream PCSX2, untouched)
```

No changes to any existing PCSX2 file. All net-new content lives at the repo root.

### Git remotes (post-setup)

```
upstream  https://github.com/PCSX2/pcsx2.git           (untouched)
origin    https://github.com/markrpearce96/pcsx2-libretro.git  NEW
```

`main` tracks `origin/main`. Rebases pull from `upstream/master` as today.

### CI workflow — `.github/workflows/libretro_release.yml`

Trigger:
```yaml
on:
  push:
    tags: ['v*']
```

**Job 1: `build`** on `macos-13` (Intel runner)
- Checkout with `submodules: recursive`
- Install Homebrew dependencies (full list deferred to the implementation plan — derived from the user's local x86_64 toolchain setup: `shaderc`, `rapidyaml`, `plutosvg`, `plutovg`, `libpng`, `libzip`, `qt`, `sdl3`, plus standard PCSX2 deps)
- Configure: `cmake -S . -B build -DENABLE_LIBRETRO=ON -DENABLE_QT_UI=OFF -DCMAKE_BUILD_TYPE=Release`
- Build: `cmake --build build --target pcsx2_libretro -j`
- Upload artifacts:
  - `pcsx2_libretro.dylib` (from `build/pcsx2-libretro/`)
  - `pcsx2_libretro_resources/` (from `bin/resources/` at the repo root — source-tracked, ~10 MB; includes `GameIndex.yaml`, `RedumpDatabase.yaml`, `Metal*.metallib`, `fonts/`, `fullscreenui/`, `icons/`, `game_controller_db.txt`)

**Job 2: `release`** (needs: `build`)
- Download both artifacts into a staging dir
- Generate `VERSION` file (format below)
- Create `pcsx2_libretro-macos-x86_64.zip` containing the dylib, resources dir, and VERSION
- Use `softprops/action-gh-release@v2` to:
  - Create the release for the pushed tag
  - Attach the zip as a release asset
  - Auto-generate release notes from commits since the previous tag, prepended with the custom prelude

### `VERSION` file format

Plain `key=value`, one per line, top-level inside the zip:

```
tag=v2026.05.21
platform=macos-x86_64
fork_sha=0751594fa...
upstream_sha=dead00eb62a7ca...
upstream_ref=upstream/master
build_date=2026-05-21T14:32:18Z
```

Rationale: a future RetroNest installer can parse this with stdlib string-split, no YAML/JSON dependency needed. `tag` is the primary staleness key; `build_date` is the secondary tie-breaker for same-day re-cuts.

### Release notes prelude

Injected by `softprops/action-gh-release@v2`'s `body` input, computed in a workflow step:

```
**Platform:** macOS x86_64 (Intel + Apple Silicon via Rosetta)
**Upstream PCSX2:** <upstream_sha_short> (master)

## Changes since previous release
<auto-generated commit list>
```

The auto-generated commit list comes from the action's built-in changelog feature (commits between this tag and the previous one on the same branch).

## Documentation

### `README.md` (~60 lines)

Sections:
1. Project description (one paragraph): libretro core fork of PCSX2, consumed by RetroNest
2. Links to upstream PCSX2 and RetroNest
3. Releases section pointing at the Releases tab as the install source
4. Local-build quickstart (the existing cmake recipe)
5. License: GPL-3.0 inherited from PCSX2
6. Pointer to `UPSTREAM-UPDATE.md`

### `UPSTREAM-UPDATE.md`

Single-source-of-truth for the rebase + release process. Concrete steps:

```
1. git fetch upstream
2. git checkout main && git rebase upstream/master
3. Resolve conflicts (typical: 0–1 anchor lines in top-level CMakeLists.txt)
4. cmake --build build --target pcsx2_libretro -j  (local sanity build)
5. (Optional) cp build/pcsx2-libretro/pcsx2_libretro.dylib + rsync bin/resources/ into ~/Documents/RetroNest/emulators/libretro/cores/ for local smoke
6. git push origin main --force-with-lease
7. git tag v$(date +%Y.%m.%d) && git push origin v$(date +%Y.%m.%d)
   -- GitHub Actions builds + publishes the release automatically
```

Plus failure-mode notes:
- CMakeLists anchor conflict (~11/12 months trivial, 30s fix)
- 2–4×/year a PCSX2 internal API the shim depended on changed → code fix inside `pcsx2-libretro/` (minutes to hour)
- Discipline: never edit upstream files outside the single 4-line block in top-level `CMakeLists.txt`
- CI failure recovery: check workflow logs, edit `.github/workflows/libretro_release.yml` if a new brew dep is missing, commit, re-tag

## Tasks (rough breakdown for the plan)

1. Manual: create empty `github.com/markrpearce96/pcsx2-libretro` on GitHub, configure default branch as `main`
2. Add `origin` remote locally, push `main` (will be a slow ~1 GB push)
3. Write `.github/workflows/libretro_release.yml`
4. Write `README.md`
5. Write `UPSTREAM-UPDATE.md`
6. Commit all three new files to `main`, push
7. Smoke-cut tag `v2026.05.21`, watch Actions, iterate on missing deps
8. Verify the published zip downloads cleanly and contains the expected three artifacts
9. Update the `project_pcsx2_libretro_port` memory with the new repo URL + the codified update process pointing at `UPSTREAM-UPDATE.md`

## Risks

- **First CI run will likely fail on missing brew deps.** The user's local `setup-x86_64-toolchain.sh` documents the full dep set; some deps may be transitively pulled by PCSX2's own bootstrap on the user's machine but not by `brew install` from a clean GHA runner. Mitigation: iterate. Re-cutting a date tag with a `.1` suffix is cheap.
- **`macos-13` runner retirement.** GitHub will eventually retire the Intel runner image. When that happens we switch to `macos-14` with Rosetta + the x86_64 brew dance from the user's local setup. Documented inline in the workflow as a comment.
- **First push is large.** ~1 GB push to a fresh GitHub repo can be slow or hit GitHub's recommended-size limits (warns at 1 GB). If it fails we can `git push` in smaller chunks via shallow-then-deepen, but in practice GitHub accepts full PCSX2 fork pushes without intervention.
- **Tag race.** If two release tags are pushed seconds apart, the workflow's "previous release" lookup could produce odd diffs. Trivial to ignore — date tags don't collide in practice.

## Out-of-scope follow-ups (recorded so they don't get lost)

- **RetroNest `Pcsx2CoreInstaller`** — mirrors `PatchesInstaller`. Fetches `pcsx2_libretro-macos-x86_64.zip` from the latest release at app start, staleness-gated by `VERSION` file's `tag` field, extracts into the cores directory. Separate sub-project.
- **Windows port (SP11?)** — replace the macOS Metal/NSView render bridge with a HWND + D3D11/Vulkan path. Needs a Windows version of RetroNest as the consumer. Multi-week effort, not a CI tweak.
- **Native macOS arm64 build** — viable once upstream PCSX2 lands AArch64 recompilers. Until then x86_64 stays the primary.
- **Code signing / notarization** — libretro cores aren't gatekeeper-checked when loaded by a notarized host. Revisit only if RetroNest's gatekeeper posture changes.
